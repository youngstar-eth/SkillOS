# ADR-001 — TournamentPool v2.1 → v2.2 migration

## Status

**Proposed.** All four contract PRs (1–4) have been authored and reviewed against the v2.2 sprint plan; merge order is sequential per the audit-discipline cadence. Mainnet deployment of v2.2 is capital-gated (audit firm engagement, ~$30–80K) and will not occur until post-audit. Base Sepolia preview deploy is permitted at any time once all four PRs land on `main`.

## Context

v2.1 (deployed at `0x52049b812780134d2F69D6c20C2ef881D49702da` on Base Sepolia) introduced the permissionless prize-pool top-up flow (`fundPrizePool`) and the v2.0 → v2.1 receipt-SBT integration. v2.1 is in active testnet use; the production contract is not changing as part of this ADR — v2.2 is a fresh deploy at a new address.

Three forces drive v2.2:

1. **Sweepstakes-safety strengthening for audit.** The v2.1 fee-handling design routed all entry fees into a single `feeCollected[id]` accumulator, with a 70/30 dev/platform share enforced *off-chain* by the operator. An auditor reviewing the on-chain code cannot verify that split holds — the relevant guarantee lives outside the contract. v2.2 makes the split bytecode-enforced.
2. **Third-party developer ecosystem.** Phase 2 of the SkillOS roadmap unlocks permissionless tournament creation by third-party developers (not just the foundation). Each developer's attribution must be on-chain, immutable, and queryable — both for fee routing and for reputation. This requires a primitive that doesn't exist in v2.1.
3. **Audit-ready code.** The audit firm receives a single coherent v2.2 artifact rather than a series of incremental patches. Storage layout, fee semantics, and access control all in one reviewable surface.

## Decision

Ship four sequential PRs that together define v2.2:

| PR | Scope | Status |
|---|---|---|
| #49 — PR 1 | `chargeRetryFee` → `chargeEntryFee` rename; `Tournament.devAddr` field; `createTournament(devAddr)` param + non-zero validation; `EntryFeePaid` event; `TournamentCreated` event extended with `devAddr` (non-indexed) | **Merged** |
| #50 — PR 2 | `feeCollected` mapping split into `feeCollected_dev` + `feeCollected_platform`; on-chain 70/30 split inside `chargeEntryFee`; `DEV_BPS=7000`, `PLATFORM_BPS=3000`, `TOTAL_BPS=10_000` constants | **Merged** |
| #51 — PR 3 | `withdrawFees(id, to)` removed; `withdrawFeesToDev(id)` (caller-authenticated, msg.sender == `Tournament.devAddr`) and `withdrawFeesToPlatform(id)` (`onlyOwner`); `OnlyDev` error; `DevFeesWithdrawn` / `PlatformFeesWithdrawn` events | **Merged** |
| #52 — PR 4 | New contract `DevAttributionNFT` (ERC-5192 soulbound); idempotent mint per `devAddr` from `TournamentPool.createTournament`; `devNFTMinted` cache mapping; deterministic `tokenId = uint256(uint160(devAddr))` | **In review** |

PRs 5 (full-lifecycle integration tests), 6 (this ADR + audit scope), and 7 (SDK type definitions, deferred) follow.

## v2.1 → v2.2 surface diff

### Storage layout (TournamentPool)

```diff
  struct Tournament {
      address sponsor;
+     address devAddr;             // immutable post-create, MUST be non-zero
      bytes32 game;
      CycleType cycleType;
      uint64 startsAt;
      uint64 endsAt;
      uint256 prizePool;
      uint256 participationBonus;
      bool settled;
      address[] participants;
  }

  // ─── State ─────
  IERC20 public immutable USDC;
+ IDevAttributionNFT public immutable devNFT;     // bound at deploy via address-prediction
  address public trustedSigner;
+ mapping(address => bool) public devNFTMinted;   // gas cache for first-mint-per-dev
  mapping(bytes32 => Tournament) internal _tournaments;
  ...
- mapping(bytes32 => uint256) public feeCollected;
+ mapping(bytes32 => uint256) public feeCollected_dev;       // 70%, DEV_BPS
+ mapping(bytes32 => uint256) public feeCollected_platform;  // 30%, PLATFORM_BPS
```

### Public surface (TournamentPool)

| v2.1 | v2.2 | Notes |
|---|---|---|
| `constructor(IERC20, address)` | `constructor(IERC20, address, address)` | Third arg: `_devNFT` (immutable, validated non-zero) |
| `createTournament(id, game, cycle, startsAt, endsAt, prizePool, bonus)` | `createTournament(id, devAddr, game, cycle, startsAt, endsAt, prizePool, bonus)` | `devAddr` validated non-zero; first-tournament-per-dev triggers NFT mint via cache |
| `chargeRetryFee(id, player)` | `chargeEntryFee(id, player)` | Renamed; body now does atomic 70/30 split into the two buckets |
| `withdrawFees(id, to)` `[onlyOwner]` | **REMOVED** | Replaced by the two functions below |
| — | `withdrawFeesToDev(id)` | Caller-authenticated: `msg.sender == Tournament.devAddr`. Pays msg.sender. |
| — | `withdrawFeesToPlatform(id)` | `onlyOwner`. Pays msg.sender. |
| `RETRY_FEE` (constant) | `ENTRY_FEE` (constant) | Same value (1 USDC = 1_000_000) |
| — | `DEV_BPS`, `PLATFORM_BPS`, `TOTAL_BPS` (constants) | Locked at 7000/3000/10000 |
| — | `OnlyDev()` error | Returned by `withdrawFeesToDev` on auth failure |
| `feeCollected[id]` (public) | `feeCollected_dev[id]`, `feeCollected_platform[id]` (public) | Two getters replace one |
| `RetryFeePaid` event | `EntryFeePaid` event | Renamed |
| `FeesWithdrawn` event | **REMOVED** | Replaced by `DevFeesWithdrawn`, `PlatformFeesWithdrawn` |
| `TournamentCreated(id, sponsor, game, cycle, startsAt, endsAt, prize, bonus)` | `TournamentCreated(id, sponsor, game, devAddr, cycle, startsAt, endsAt, prize, bonus)` | `devAddr` added as non-indexed (3 indexed slots already saturated by id/sponsor/game) |

### New contract: DevAttributionNFT

Soulbound (ERC-5192) ERC-721 with the following surface:

- `constructor(address _tournamentPool)` — pins the bound pool via immutable; non-zero validated
- `mint(address dev)` — only callable by `tournamentPool`; non-zero `dev` validated; emits `Locked(tokenId)` per ERC-5192
- `locked(uint256 tokenId)` — returns `true` for every minted token; reverts on non-existent (per spec)
- `approve(address, uint256)` — overridden to revert `Soulbound()`
- `setApprovalForAll(address, bool)` — overridden to revert `Soulbound()`
- `_update(...)` — overridden to revert `Soulbound()` on any non-mint movement (transfers AND burns)
- `supportsInterface(bytes4)` — adds `IERC5192` to ERC-721's defaults

Deterministic `tokenId = uint256(uint160(devAddr))` makes the NFT trivially queryable by wallet without an off-chain index.

## Storage layout migration plan

**There is no migration.** v2.2 is a fresh contract deploy at a new address. v2.1 (`0x52049b...`) remains live for the testnet validation tail; v2.2 deploys to a new address determined at audit-clear time.

Implications:
- No upgrade proxy. v2.2 contracts are non-upgradeable (consistent with v2.1's design).
- v2.1 in-flight tournaments settle on v2.1; their `feeCollected` is still the legacy single mapping and is withdrawn via the legacy `withdrawFees(id, to)` (still present on the deployed v2.1 bytecode).
- Backend/cron writes the new v2.2 address into the operational config at cutover; new tournaments target v2.2 from then on.
- Off-chain indexer (`TournamentCreated` consumer) needs an update for the new event signature (additional `devAddr` field in the non-indexed payload). Coordinated separately in the SDK rename sprint or PR 7 if applicable.

## Backward compatibility analysis

**None at the contract layer** — v2.2 has no shared state with v2.1. Specifically:

- ABI: v2.2 introduces a new constructor signature, a new mapping shape (`feeCollected_dev` / `feeCollected_platform` instead of `feeCollected`), three new public functions, and renamed events. ABIs are not interchangeable.
- Storage: v2.2 includes new fields (`Tournament.devAddr`) and new mappings (`feeCollected_dev`, `feeCollected_platform`, `devNFTMinted`); v2.1 storage layout cannot be cast to v2.2 layout.
- Events: signatures differ (renamed + extended). Off-chain indexers must be updated.

The only continuity is operational: the team-treasury wallet, USDC token, sanctions oracle, and `STUDIO_PRIVATE_KEY` signer carry over unchanged. Sponsorship integration (`SponsorshipModule` + `SponsorReceiptSBT`) is unchanged in behavior; their constructor args were not modified.

## Risk register (for audit firm)

| ID | Risk | Mitigation | Residual |
|---|---|---|---|
| R1 | Storage segregation between fee buckets and prize pool fails — entry fees leak into prize distribution or vice versa | INV1 invariant tests pin every code path; `feeCollected_*` and `t.prizePool` live on disjoint keccak-derived storage slots; no function reads cross-bucket | Low — pinned by ~6 invariant tests |
| R2 | 70/30 atomic split introduces dust at non-locked future ENTRY_FEE values | At locked constants (`ENTRY_FEE = 1_000_000`, `DEV_BPS = 7000`, `PLATFORM_BPS = 3000`, `TOTAL_BPS = 10_000`), `gcd(7000, 3000, 10_000) = 1000` so the dust-free domain is `{ENTRY_FEE : ENTRY_FEE % 10 == 0}`. Current value satisfies. The `testFuzz_chargeEntryFee_noDust_holdsForFutureEntryFees` and `test_chargeEntryFee_noDust_offBy1_introducesDust` tests document this domain explicitly | Acceptable — any future value change requires the test domain to be re-pinned |
| R3 | `devNFTMinted` cache desyncs from `DevAttributionNFT` state (e.g., NFT redeployed without resetting cache) | Cache flipped BEFORE external `mint()` (CEI). OZ `_safeMint` reverts on duplicate `tokenId` — surfaces any cache desync as a hard revert in `createTournament`, not silent corruption | Low — defensive backstop in place |
| R4 | Compromised `devAddr` key drains the dev fee bucket via `withdrawFeesToDev` | By design: `withdrawFeesToDev` pays `msg.sender` (the auth identity). A compromised key drains to that key, not an attacker's wallet — drain is then traceable on-chain. Funds don't leave the dev's address space without a separate transfer the attacker would also need to initiate from the same key | Acceptable — identity == destination is the deliberate design, not a hole |
| R5 | Permissionless `createTournament` flooding to mint many `DevAttributionNFT`s and consume gas | Each mint costs ~78K gas to the *caller of createTournament* (the sponsor depositing the prize pool). Sponsor pays USDC for the prize, so flooding has positive cost to attacker; not a viable DoS surface | Acceptable — economically gated |
| R6 | First-tournament-per-dev gas cost (~78K) is too high for the target UX | Per-feature cost, not a regression. Subsequent tournaments hit the cache (~2K SLOAD only). Documented in PR 4. If material, defer to a follow-up optimization PR (see "Open questions" below) | Acceptable — visible at deploy time |
| R7 | Sanctions oracle integration not exercised in v2.2 changes | Out of scope for v2.2. SponsorshipModule continues to enforce sanctions on the sponsor-pool funding path; production swap of the mock oracle is a separate sprint | Acceptable — orthogonal concern |
| R8 | Dev attribution NFT minted to incorrect address due to off-chain devAddr typo | Tests `test_createTournament_storesDevAddr` and `test_createTournament_revert_zeroDevAddr` pin the recorded address. Off-chain validation by the SDK is the upstream defense | Operational, not contract |

## Open questions (founder + audit)

- **Q1 (deferred from PR 4):** Is the ~78K gas first-mint cost acceptable for the target UX? If not, optimization options include: (a) lazy mint via `claimDevNFT()`, (b) skip `_safeMint` (use plain `_mint`) for ~5K savings, (c) accept current. Recommend deferring decision until first audit-firm pass and a real cost-benefit reading.
- **Q2 (deferred from PR 2 review N5):** Should `t.prizePool` be zeroed after `settle` for view-layer accuracy? Reviewer concluded it's view-layer staleness, not safety. Founder decision pending — fits naturally into PR 5 integration tests.
- **Q3:** Should `withdrawFeesToPlatform` accept a `to` parameter for treasury routing? Current single-arg design pays `msg.sender == owner()`. Operationally the owner can transfer externally, or the owner address can be set to a treasury multisig at deploy time. Audit firm may flag the UX cost.
